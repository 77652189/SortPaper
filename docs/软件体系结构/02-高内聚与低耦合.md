# 02-高内聚与低耦合

你写 RAG 或 Agent 项目时，最常见的失控不是算法不够高级，而是“东西放错地方”。一个模块本来只该负责解析 PDF，后来顺手做了 chunk 清洗、质量判断、入库、日志、UI 提示。短期看省事，长期看每次改动都会牵动一串。

高内聚和低耦合就是用来回答：哪些代码应该放在一起，哪些代码必须分开。

## 为什么需要它

高内聚：一个模块里的代码应该围绕同一种责任、同一种变化原因聚在一起。

低耦合：一个模块不应该知道太多别的模块内部细节。

在 AI 辅助开发里，这两个原则更重要。因为 Codex 修改代码时会根据上下文推断意图。如果一个文件里混着 UI、业务规则、数据库和 LLM 调用，它就很难判断哪些地方能改、哪些地方不能碰。

## 什么时候需要重点检查

需要：

- 一个函数超过几十行，并且出现多个阶段名，比如 parse、clean、store、judge、render。
- 一个模块 import 了很多外部服务，比如 qdrant、openai、streamlit、pdf parser。
- 同一段业务规则在多个地方重复，比如 chunk 是否可入库的判断。
- 新增功能时总要修改 3 个以上不相关文件。

暂时可以放松：

- 原型验证阶段，还没有确认业务流程是否成立。
- 只有一个实现、一个调用方，拆开反而增加理解成本。
- 模块虽然长，但所有代码都围绕同一个变化原因，比如一个复杂表格清洗算法。

## 例子：论文解析流程

坏设计：

```text
process_pdf(file):
  从 UI 读取选项
  解析 PDF 文本
  解析表格
  调用 LLM 判断 chunk 质量
  写入 Qdrant
  更新 session_state
  渲染结果
```

它的问题不是“函数长”，而是变化原因太多：

- UI 选项变化会影响它。
- PDF 解析策略变化会影响它。
- Judge prompt 变化会影响它。
- Qdrant payload 变化会影响它。
- 展示方式变化也会影响它。

更好的拆法：

```text
PdfParser: PDF -> raw blocks
ChunkBuilder: raw blocks -> chunks
ChunkJudge: chunks -> verdicts
ChunkStorage: chunks + verdicts -> store stats
ResultPresenter: result -> UI view model
```

这些模块可以串起来，但不要互相钻进对方内部。

## 例子：长期记忆写入

坏设计：

```text
answer_question(query):
  检索论文
  生成回答
  如果用户喜欢长回答，就改 prompt
  如果出现偏好，就写入记忆
  如果像风格样本，就写入风格库
```

这里把“回答问题”和“学习用户”混在一起。以后你想禁用风格学习，可能会影响问答；你想调整记忆写入阈值，也要碰回答逻辑。

更好的拆法：

```text
AnswerService: query + context -> answer
MemoryCandidateExtractor: interaction -> memory candidates
MemoryPolicy: candidate -> should write?
StyleLearner: answer/user edits -> style sample
```

回答系统可以触发学习，但不应该包含学习规则本身。

## 坏设计长什么样

- 一个模块名字很泛，比如 `utils`、`pipeline`、`manager`，里面什么都有。
- 数据结构在多个模块里被就地修改，调用方不知道谁改了字段。
- 模块之间传大字典，字段没有契约。
- A 模块为了方便直接调用 B 模块内部函数，而不是通过公开接口。
- 测试只能 mock 半个世界才能跑。

## 更好的拆法

先问三个问题：

1. 这个模块的主要产物是什么？
2. 哪类需求变化会迫使它修改？
3. 它必须知道哪些外部细节，哪些只是临时方便？

如果一个模块有两个以上主要产物，通常应该拆。

如果一个模块会因为 UI、数据库、LLM prompt 三种变化而修改，通常应该拆。

如果一个模块需要知道另一个模块的数据字段细节，应该考虑定义明确的输入输出类型。

## 可执行产物：模块拆分判断标准

```markdown
## 模块拆分检查

模块名称：

### 主要职责
- 它负责产出什么？
- 它不负责什么？

### 输入输出
- 输入数据是什么？
- 输出数据是什么？
- 是否存在隐式依赖，比如全局变量、session_state、环境变量？

### 变化原因
- UI 变化会不会改它？
- 数据库变化会不会改它？
- LLM prompt 变化会不会改它？
- 业务规则变化会不会改它？

### 拆分判断
- 如果变化原因超过 2 类，考虑拆分。
- 如果输出超过 1 类核心产物，考虑拆分。
- 如果测试需要启动 UI 或真实外部服务，考虑抽接口。
```

## 坏味道清单

看到这些名字和行为时要警惕：

- `process_all`
- `run_everything`
- `handle_request_and_store`
- `parse_and_save_and_render`
- `global_config` 被到处读取。
- `dict` 里字段靠约定，没有文档。
- 一个函数既返回业务结果，又返回 UI 文案。

## 给 Codex 的提示词

```text
请按高内聚、低耦合原则检查这个模块。
不要只说文件太长，要指出每段代码的变化原因。
请输出：
1. 当前模块的职责列表
2. 哪些职责应该留在一起
3. 哪些职责应该拆出去
4. 拆分后建议的接口输入输出
先不要修改代码。
```
