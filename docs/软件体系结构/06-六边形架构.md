# 06-六边形架构

六边形架构听起来抽象，但它解决的问题很日常：不要让核心业务被外部技术绑死。

你做 RAG 时，核心业务不是 Qdrant；你做 Agent 时，核心业务不是某个 LLM SDK；你做长期记忆时，核心业务不是向量数据库。外部技术会换，业务规则最好不要跟着碎掉。

## 为什么需要它

六边形架构把系统分成两类东西：

- 内部核心：业务规则、用例、状态变化。
- 外部适配器：数据库、LLM、文件系统、Web 框架、第三方 API。

内部通过 Port 描述自己需要什么能力。外部通过 Adapter 实现这些能力。

```text
核心业务 -> Port 接口 <- Adapter 实现
```

这样核心业务只知道“我需要检索相似 chunk”，不需要知道“这是 Qdrant 的 search API，payload 长这样”。

## 什么时候需要

需要：

- 外部技术未来可能更换，比如 FAISS 换 Qdrant，OpenAI 换本地模型。
- 你想对核心业务写单元测试，不想启动真实外部服务。
- 同一能力有多个实现，比如多个 PDF parser、多个 reranker。
- Codex 经常因为 SDK 细节修改核心流程。

暂时不需要完整六边形：

- 外部依赖非常少，且短期不会更换。
- 项目还在快速试错，接口稳定性不够。
- 抽接口只是复制实现，没有降低复杂度。

## 例子：RAG 检索 Port

坏设计：

```text
RagService:
  qdrant_client.search(collection="papers", vector=embedding, limit=10)
  解析 Qdrant payload
  拼接上下文
```

`RagService` 直接依赖 Qdrant。以后换存储或调整 payload，RAG 核心流程也要跟着改。

更好的设计：

```text
Port:
  ChunkRetriever.search(query, filters) -> list[RetrievedChunk]

Adapter:
  QdrantChunkRetriever
  InMemoryChunkRetriever
  HybridChunkRetriever

Core:
  RagService 只依赖 ChunkRetriever
```

测试时可以用 `InMemoryChunkRetriever`，真实运行时用 `QdrantChunkRetriever`。

## 例子：长期记忆 Store

坏设计：

```text
MemoryService:
  生成 embedding
  写 Qdrant
  读 Qdrant
  解析 payload
  判断记忆冲突
```

这里核心记忆策略和存储细节混在一起。

更好的设计：

```text
Port:
  MemoryStore.save(memory)
  MemoryStore.search(user_id, query)
  MemoryStore.mark_obsolete(memory_id)

Adapter:
  VectorMemoryStore
  SqlMemoryStore
  FileMemoryStore

Core:
  MemoryPolicy
  MemoryUpdateService
```

核心规则关心“该不该记、怎么合并、何时过期”，不关心具体数据库。

## 例子：Agent Tool Port

Agent 工具也适合用 Port：

```text
Tool:
  name
  input_schema
  execute(input) -> ToolResult
```

Agent 核心只依赖统一工具协议，不直接知道每个工具背后的数据库、HTTP API 或文件系统。

## 坏设计长什么样

- 核心服务直接 import Qdrant、OpenAI SDK、Streamlit。
- 测试核心业务时必须连真实外部服务。
- 替换一个 provider 要改很多业务文件。
- 外部 API 返回结构直接在系统内部到处流动。
- prompt、payload、数据库字段和业务规则绑在一起。

## 更好的拆法

- 先从核心业务视角命名接口，不从技术视角命名。
- Port 返回系统内部稳定对象，不返回外部 SDK 原始对象。
- Adapter 负责翻译外部格式和内部格式。
- 核心服务只依赖 Port，不依赖 Adapter。
- 配置决定使用哪个 Adapter。
