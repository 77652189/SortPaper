# 07-插件化架构

插件化架构解决的是“未来会有很多种实现，但主流程不应该跟着膨胀”的问题。

在论文解析和 RAG 系统里，你很容易遇到这种情况：今天支持一个 PDF parser，明天要加表格 parser，后天要加图片 parser；今天用一个 reranker，后面要对比多个 reranker；Agent 工具也会越来越多。每新增一种实现都改主流程，系统会越来越难维护。

## 为什么需要它

插件化的核心不是“动态加载很高级”，而是把扩展点固定下来。

主流程只知道：

```text
给我一个符合协议的解析器
给我一个符合协议的检索器
给我一个符合协议的 Agent 工具
```

它不应该写一长串：

```text
if parser_type == "pymupdf":
elif parser_type == "marker":
elif parser_type == "ocr":
elif parser_type == "table_only":
```

插件化能让新增能力变成“注册一个实现”，而不是“改主流程”。

## 什么时候需要

需要：

- 同一能力已经有 2 种以上实现，且未来还会增加。
- 不同项目或不同配置需要启用不同能力。
- 你想让 Codex 新增插件时，不碰核心流程。
- 解析器、检索器、记忆策略、工具之间有统一协议。

暂时不要用：

- 只有一个实现，而且未来不确定是否扩展。
- 插件协议还没想清楚，先抽象会很空。
- 插件需要共享大量内部状态，说明边界还没切好。

## 例子：PDF 解析器插件

坏设计：

```text
parse_pdf(file, mode):
  if mode == "text":
    parse_text()
  elif mode == "table":
    parse_table()
  elif mode == "image":
    parse_image()
  elif mode == "ocr":
    parse_ocr()
```

每新增一个解析模式都要改主函数。

更好的设计：

```text
ParserPlugin:
  name
  supports(file, config)
  parse(file, config) -> list[Block]

Registry:
  register(TextParser)
  register(TableParser)
  register(ImageParser)

Pipeline:
  selected_plugins = registry.resolve(config)
  run each plugin
```

主流程不关心每个插件内部怎么做，只关心它是否符合协议。

## 例子：Agent 工具插件

Agent 工具天然适合插件化：

```text
ToolPlugin:
  name
  description
  input_schema
  execute(input, context) -> ToolResult
```

论文问答 Agent 可以注册：

- `search_literature`
- `summarize_paper`
- `compare_methods`
- `inspect_memory`
- `update_style_profile`

Agent 核心只通过工具协议调用，不直接依赖每个工具实现。

## 例子：记忆压缩策略插件

长期记忆系统可能有多种压缩策略：

- 保留最近对话。
- 合并相似偏好。
- 生成用户画像摘要。
- 删除过期事实。

这些适合设计成策略插件：

```text
MemoryCompactionStrategy:
  compact(memories, context) -> CompactionResult
```

## 坏设计长什么样

- 主流程里有大量根据类型分支的 if / elif。
- 新增一种实现必须修改核心 pipeline。
- 插件之间直接互相调用，绕过注册中心。
- 插件返回格式各不相同，主流程到处做兼容。
- 配置项和插件名散落在多个文件。

## 更好的拆法

- 先定义插件协议：输入、输出、错误、元数据。
- 建立注册中心，集中管理可用插件。
- 主流程只依赖协议和注册中心。
- 插件返回统一内部对象，不返回各自私有格式。
- 插件配置集中声明，避免散落 if。

## 可执行产物：插件最小方案

```markdown
## 插件设计

扩展点名称：

### 为什么需要插件化
- 当前实现数量：
- 未来可能新增：
- 主流程不应该知道的细节：

### 插件协议
- name：
- config_schema：
- supports：
- run / execute：
- result：
- error：

### 注册方式
- 静态注册：
- 配置启用：
- 默认插件：

### 主流程规则
- 主流程只调用协议方法。
- 主流程不根据具体插件名写业务分支。
- 插件失败必须返回统一错误形状。
```

```text
1. 当前硬编码分支
2. 建议的插件协议
3. 注册方式
4. 新增一个插件时不应修改哪些核心文件
先不要修改代码。
```
