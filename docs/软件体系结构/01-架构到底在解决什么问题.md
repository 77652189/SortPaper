# 01-架构到底在解决什么问题

很多项目不是一开始就需要“架构”。一个脚本能跑、一个页面能展示、一个接口能返回结果时，先做出来通常是对的。真正的问题出现在第二阶段：你开始加解析、加缓存、加向量库、加 Agent、加长期记忆、加风格学习，代码还保持“先能跑”的形状。

这时架构要解决的不是“怎么画图”，而是四件事：

- 复杂度：功能越来越多，人还能不能理解。
- 变化：换模型、换向量库、换解析器时，改动会不会扩散。
- 协作：你和 Codex 或其他人能不能在不互相踩线的情况下改代码。
- 风险：失败、重试、降级、脏数据会不会污染核心流程。

## 为什么需要它

没有架构时，项目常常还能继续加功能，但每一步都变慢。

在论文解析系统里，最初也许只是：

```text
上传 PDF -> 解析文本 -> 切 chunk -> 写入向量库 -> 问答
```

后来你会加入：

- 表格解析和图片解析。
- LLM Judge 判断 chunk 质量。
- Qdrant payload 回写质量分析。
- Agent 根据用户问题多轮检索。
- 长期记忆记录用户偏好。
- 风格学习让回答更像某种写作习惯。

如果这些逻辑都塞在一个“主流程函数”里，问题会很快出现：你想调整 chunk 清洗，却影响了入库；你想换 embedding provider，却牵动 UI；你让 Codex 改 Agent 工具，它顺手改了检索 payload。

架构的价值就是提前放置一些边界，让变化被限制在合理范围内。

## 什么时候需要架构设计

需要：

- 一个功能已经有 3 个以上外部依赖，比如 LLM、向量库、文件系统、Web UI。
- 同一个数据会经过多个阶段，比如 PDF bytes、raw text、chunk、embedding、retrieved context、answer。
- 同一套流程有多种模式，比如 preview、full pipeline、batch import、agent query。
- 你已经发现“改 A 经常坏 B”。
- 你准备让 Codex 做超过一个文件的改动。

暂时不需要复杂架构：

- 只是一次性脚本。
- 功能还在验证可行性，接口和数据形状都不稳定。
- 只有一个调用方、一个实现、一个数据源。
- 失败后手动重跑成本很低。

这里的重点是“不需要复杂架构”，不是“不需要边界”。即使是脚本，也可以保留最基本的函数输入输出清晰。

## 例子：RAG 检索链路

坏情况：

```text
search(query):
  生成 query embedding
  访问 Qdrant
  rerank
  拼 prompt
  调 LLM
  写日志
  更新用户长期记忆
  返回 UI 展示对象
```

这个函数看起来方便，但它把检索、生成、记忆和展示混在一起。Codex 想优化 rerank 时，可能不小心改变 prompt；你想禁用长期记忆，也要进入主检索函数里拆线。

更好的拆法：

```text
RagApplicationService
  -> QueryUnderstanding
  -> Retriever
  -> Reranker
  -> ContextBuilder
  -> AnswerGenerator
  -> MemoryRecorder
```

每个模块都只回答一个问题。比如 `Retriever` 只负责“从知识库拿候选内容”，不负责“最终怎么回答”。

## 例子：Agent 工具调用失控

坏情况：

```text
Agent loop 里直接访问数据库、文件系统、Qdrant、LLM、配置文件。
每个工具返回什么格式都不一样。
工具失败后有的抛异常，有的返回空字符串，有的把错误塞进 answer。
```

这种系统很难排错。Agent 失败时，你不知道是模型计划错了、工具输入错了、检索没命中，还是记忆污染了上下文。

更好的拆法：

```text
AgentController
  -> PlanState
  -> ToolRegistry
  -> ToolExecutor
  -> ObservationNormalizer
  -> FailurePolicy
```

这不是为了“显得复杂”，而是为了让 Agent 的每一步可观察、可重试、可限制。

## 坏设计长什么样

- 一个文件同时负责 UI、业务流程、数据库、LLM 调用。
- 一个函数参数里塞了很多字典，没人知道哪些字段必须存在。
- 数据形状在流程中不断被临时修改，字段语义漂移。
- 测试只能测完整流程，无法单独测解析、检索、记忆。
- Codex 每次改动都需要读很多无关文件。

## 更好的拆法

- 先找数据流：输入是什么，中间产物是什么，输出是什么。
- 再找变化点：哪些东西未来可能换，哪些规则相对稳定。
- 把稳定规则放核心，把容易替换的技术放边缘。
- 用接口描述边界，不让调用方依赖具体实现细节。
- 为每条长流程建立状态和失败处理，而不是靠散落的 if。

## 可执行产物：项目是否需要架构设计检查表

把下面问题复制到项目文档里，逐项回答。

```markdown
## 架构复杂度检查

- 当前流程是否超过 5 个步骤？
- 是否依赖 LLM、向量库、文件系统、数据库、UI 框架中的 2 个以上？
- 是否存在 preview / full / batch / agent 等多种模式？
- 是否有多个模块读写同一种数据结构？
- 是否存在“字段名一样但含义不同”的情况？
- 是否需要失败重试、降级、跳过、恢复？
- 是否计划让 Codex 做跨文件改动？
- 是否有功能经常因为无关改动而坏掉？

如果 3 项以上为“是”，至少需要画出模块边界和数据流。
如果 5 项以上为“是”，需要写接口契约和测试计划。
```

## 给 Codex 的提示词

```text
请从架构角度审查当前模块。
重点不是代码风格，而是复杂度、变化点、职责边界和测试边界。
请输出：
1. 当前数据流
2. 主要职责混杂点
3. 最小可行拆分方案
4. 每一步拆分的风险
先不要修改代码。
```
