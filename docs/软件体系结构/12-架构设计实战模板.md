# 12-架构设计实战模板

这一章不是用来阅读的，而是用来复制的。每次你要做一个中等以上功能，比如 RAG 入库重构、Agent 工具系统、长期记忆、风格学习，都可以先填这份模板，再交给 Codex 实现。

模板的目标是让实现者不需要猜：目标、边界、数据流、接口、失败处理、测试标准都写清楚。

## 为什么需要它

架构设计最容易漏掉的不是“模块名”，而是这些问题：

- 输入输出到底是什么。
- 哪些行为必须保持兼容。
- 哪些外部技术只是实现细节。
- 失败时应该返回什么。
- 怎么判断做完了。

把这些写清楚，Codex 才能稳定执行。

## 什么时候使用

需要：

- 功能会跨 2 个以上模块。
- 涉及 RAG、Agent、长期记忆、风格学习这类长流程。
- 需要保持现有入口兼容。
- 需要让 Codex 先理解边界再实现。
- 改动失败会污染数据、记忆或索引。

暂时不需要：

- 只是改一个局部 bug。
- 只是调整文案、样式或简单配置。
- 功能还在一次性原型阶段，数据结构很快会推翻。

## 例子：RAG 入库重构

不要只写：

```text
优化入库逻辑。
```

应该写清：

- 保持 UI 调用入口不变。
- `raw_content`、`paper_id`、`chunk_id` 语义不能变。
- clean / degraded / excluded 三类 chunk 的规则要集中到一个 service。
- 先补规则测试，再移动实现。

这样 Codex 不会把“优化”理解成重写整条 RAG pipeline。

## 例子：长期记忆最小版本

不要只写：

```text
加长期记忆。
```

应该写清：

- 只实现 preference 和 project 两类记忆。
- 写入必须经过 `MemoryPolicy`。
- 先用内存版 `MemoryStore`，不要直接接向量库。
- 检索结果必须带来源、类型、置信度。

这样可以先验证记忆边界，再决定是否接入复杂存储。

## 坏设计长什么样

- 文档只写“实现某功能”，没有说明边界。
- 没有列出保持不变的接口和数据字段。
- 没有失败处理，默认所有外部服务都会成功。
- 没有测试计划，完成标准只能靠主观感觉。
- 把 RAG、Agent、记忆、风格学习一次性塞进同一个任务。

## 更好的拆法

- 先写目标和不做什么。
- 再写数据流和模块职责。
- 然后写接口、错误、兼容性。
- 最后写测试和 Codex 执行顺序。
- 大功能拆成多个小任务，每个任务只移动一个边界。
