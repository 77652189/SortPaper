# 软件体系结构速成课程包

这套课程包不是为了背概念，也不是为了画漂亮的架构图。它的目标很直接：让你在做 RAG、论文解析、Agent、长期记忆、风格学习系统时，能判断代码应该怎么拆、边界应该怎么定、哪些改动可以放心交给 Codex，哪些地方必须先写清楚契约。

如果你经常遇到这些问题，这套内容就是为你准备的：

- 一个功能开始只是脚本，后来变成了很多 if、很多状态、很多外部服务。
- UI、解析、LLM 调用、向量库写入、质量评估都混在一起，想改一个地方会牵动一串。
- 想让 Codex 帮你重构，但不知道如何描述目标边界，结果它容易改大了。
- 听过高内聚、低耦合、MVC、分层、六边形，但不知道它们和 RAG / Agent 项目有什么关系。

## 怎么使用

建议先读：

1. `00-学习路线.md`
2. `01-架构到底在解决什么问题.md`
3. `02-高内聚与低耦合.md`
4. `04-分层架构.md`
5. `08-RAG系统架构.md`
6. `09-Agent系统架构.md`
7. `11-AI辅助开发中的架构协作.md`

如果你正在改真实项目，不要一次读完。更好的方式是：读一章，拿当前项目对照一遍，把能改的小边界先写出来，再让 Codex 做小步重构。

## 学习原则

- 先问“为什么需要这个架构”，再看“它叫什么名字”。
- 先识别变化点，再决定模块边界。
- 先写接口和输入输出，再让 AI 生成实现。
- 先让功能可测试，再追求设计漂亮。
- 架构不是越复杂越好，能解释变化、隔离风险、方便协作才算好。

## 和 Codex 协作的方式

读完每章后，可以把该章的模板直接贴给 Codex，让它帮你做一次局部检查。例如：

```text
请按 docs/软件体系结构/02-高内聚与低耦合.md 的检查表，
审查当前 RAG 入库流程中哪些函数职责过多。
只输出问题清单和建议拆分，不要修改代码。
```

当你准备实现时，再把任务变成更具体的指令：

```text
请把解析结果入库逻辑从应用编排层拆到独立 storage service。
要求保持现有 UI 调用入口不变，先补测试，再改实现。
```
