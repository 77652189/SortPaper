# 11-AI 辅助开发中的架构协作

和 Codex 协作时，最重要的不是把需求写得很长，而是把边界写得清楚。AI 很擅长补全实现，但如果你没有告诉它哪些接口不能变、哪些层不能碰、哪些行为必须保留，它会用“看起来合理”的方式扩大改动。

架构文档的价值，是把隐含判断变成可执行约束。

## 为什么需要它

AI 辅助开发里的常见失败：

- 让它“优化一下”，结果它重写了半个模块。
- 让它“加长期记忆”，结果直接把聊天记录塞进现有向量库。
- 让它“改 RAG 检索”，结果 prompt、入库、UI 一起改。
- 让它“重构架构”，结果没有测试边界，行为悄悄变了。

好的任务说明要让 Codex 知道：

- 目标是什么。
- 成功标准是什么。
- 哪些文件或接口必须保持兼容。
- 哪些职责应该被拆开。
- 改完怎么验证。

## 什么时候先写架构任务说明

需要：

- 改动跨多个文件。
- 涉及数据结构、接口、存储、外部服务。
- 需要保持现有行为不变。
- 需要让 Codex 先审查再改。
- 你不确定当前边界是否合理。

可以直接让 Codex 实现：

- 单个函数的小 bug。
- 文案、注释、简单配置。
- 明确局部测试覆盖的小改动。

## 例子：给 RAG 入库流程的任务

模糊指令：

```text
帮我优化 RAG 入库流程。
```

更好的指令：

```text
请把 RAG 入库流程中的 chunk 存储规则从应用编排层拆到独立 service。
保持现有 UI 调用入口不变。
不要改变 Qdrant payload 字段语义。
先补针对 clean / degraded / excluded 三类 chunk 的测试，再改实现。
改完运行相关测试，并总结行为是否变化。
```

这个指令告诉 Codex：哪里能改，哪里不能改，怎么验收。

## 例子：给 Agent 工具系统的任务

模糊指令：

```text
把 Agent 工具做得更规范。
```

更好的指令：

```text
请为 Agent 工具定义统一 Tool 协议，包括 name、description、input_schema、execute、ToolResult。
先只迁移 search_literature 工具。
保持 Agent 对外调用方式不变。
工具失败时返回结构化错误，不要抛到 UI。
新增测试覆盖成功、空结果、工具异常三种情况。
```

## 例子：给长期记忆功能的任务

模糊指令：

```text
加一个长期记忆。
```

更好的指令：

```text
请先实现长期记忆的最小架构，不直接接入所有聊天记录。
新增 MemoryCandidate、MemoryPolicy、MemoryStore Port。
只支持 preference 和 project 两类记忆。
写入必须经过 MemoryPolicy 判断。
检索结果必须带 memory_id、type、confidence、source。
先用内存实现 MemoryStore，后续再接向量库。
```

## 坏设计长什么样

- 任务只描述“想要什么”，没有描述“不允许破坏什么”。
- 没有测试标准，完成与否靠感觉。
- 让 Codex 一次性改太多层。
- 让 Codex 在没有接口契约的情况下生成存储结构。
- 重构任务没有要求保持现有入口兼容。

## 更好的拆法

- 先让 Codex 读代码并输出现状，不急着改。
- 把任务切成接口、测试、实现、清理四步。
- 每次只改一个边界。
- 对外入口保持不变，内部逐步替换。
- 用检查清单约束架构，而不是靠口头感觉。

## 可执行产物：Codex 架构任务模板

```markdown
## Codex 任务说明

### 目标
- 这次要实现或重构什么：
- 为什么需要：

### 保持不变
- 公开 API：
- UI 行为：
- 数据字段语义：
- 兼容性要求：

### 允许改变
- 可以新增的模块：
- 可以移动的职责：
- 可以重命名的内部函数：

### 架构边界
- 属于入口层的逻辑：
- 属于应用层的逻辑：
- 属于领域层的逻辑：
- 属于基础设施层的逻辑：

### 验收标准
- 测试：
- 手动验证：
- 不应出现的回归：

### 执行顺序
1. 先审查现状。
2. 补或调整测试。
3. 小步实现。
4. 运行验证。
5. 总结改动和残余风险。
```

## 给 Codex 的提示词

```text
请根据上面的任务说明执行。
如果发现现有代码和任务说明冲突，先报告冲突，不要自行扩大范围。
实现时保持公开入口兼容。
完成后请说明测试结果、行为变化和未覆盖风险。
```
