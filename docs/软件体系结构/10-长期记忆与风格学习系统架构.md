# 10-长期记忆与风格学习系统架构

长期记忆最危险的误解是：把聊天记录全部塞进向量库，就叫有记忆。这样做很快，但后面会遇到一堆问题：过期信息被召回、临时情绪变成长期偏好、重复记忆越来越多、风格样本和事实记忆混在一起。

长期记忆和风格学习系统要解决的是“什么值得长期保留，什么时候使用，什么时候忘掉”。

## 为什么需要它

没有架构的记忆系统会让 AI 看起来越来越“了解你”，但实际越来越不可控：

- 用户临时说“今天不要长回答”，系统永久记住。
- 一个偏好被重复写入很多次。
- 旧项目的上下文污染新项目。
- 风格样本被当成事实使用。
- 回答中用了记忆，但没有记录来源。

记忆系统必须有写入规则、检索规则、冲突处理和遗忘机制。

## 记忆类型

| 类型 | 例子 | 使用方式 |
|---|---|---|
| 事实记忆 | 用户正在做 PaperSort 项目 | 用于上下文补全 |
| 偏好记忆 | 用户希望中文回答 | 用于回答策略 |
| 项目记忆 | 当前项目使用 Qdrant 和 Streamlit | 用于开发协作 |
| 交互记忆 | 上次讨论过某个重构方案 | 用于连续任务 |
| 风格样本 | 用户偏好的表达方式、文档口吻 | 用于生成风格 |

风格学习要和事实记忆分开。风格样本说明“怎么表达”，事实记忆说明“世界是什么”。

## 什么时候需要

需要：

- 系统会长期和同一用户协作。
- 用户偏好会影响回答、文档、代码风格。
- Agent 需要跨会话延续项目上下文。
- 系统会从用户修改中学习风格。
- 记忆错误会明显影响结果质量。

暂时不要做复杂记忆：

- 只是一次性问答。
- 用户不希望系统保留长期信息。
- 没有记忆审查和删除能力。
- 写入规则不清楚，容易记错。

## 例子：AI 辅助开发记忆

值得记：

- 用户要求“用中文回答”。
- 当前项目关注 RAG、论文解析、Agent、长期记忆、风格学习。
- 用户偏好先做计划再实现复杂功能。

不应该长期记：

- 某次临时调试中的错误日志。
- 用户当天情绪化说“以后都别写测试”。
- 某个过期分支的路径。

记忆系统要区分稳定偏好和临时上下文。

## 例子：风格学习

风格学习不是记住每一句话，而是提取可复用模式：

- 标题短，不写教材腔。
- 先讲为什么，再讲怎么做。
- 输出适合保存为 Markdown。
- 给 Codex 的任务要有边界和验收标准。

风格样本可以来自用户修改后的文档。系统要保存“原始输出 -> 用户修改 -> 提炼出的风格规则”，而不是只保存最终文本。

## 例子：RAG 和记忆一起工作

回答一个项目问题时，系统可能需要两类上下文：

- 知识库：项目文档、代码、论文内容。
- 长期记忆：用户偏好、当前目标、之前决策。

坏设计是把两类内容混在一个向量库里不加区分。更好的方式是：

```text
KnowledgeContext:
  来自项目和论文资料

MemoryContext:
  来自用户偏好和历史决策

ContextBuilder:
  明确标注来源和可信度
```

## 坏设计长什么样

- 所有聊天记录无条件写入。
- 检索出的记忆没有类型、时间、置信度。
- 风格样本和事实记忆混放。
- 没有删除、过期、合并机制。
- Agent 不记录本轮用了哪些记忆。
- prompt 里直接塞入大量记忆，导致上下文污染。

## 更好的拆法

- 写入前先生成 `MemoryCandidate`。
- 用 `MemoryPolicy` 判断是否值得写。
- 每条记忆保存类型、来源、时间、置信度、适用范围。
- 检索后再过滤，不是召回什么都用。
- 风格学习独立保存样本和规则。
- 定期合并重复记忆，标记过期记忆。

## 给 Codex 的提示词

```text
请为当前系统设计长期记忆和风格学习模块。
要求区分事实记忆、偏好记忆、项目记忆、交互记忆、风格样本。
请输出写入条件、检索条件、遗忘策略、冲突处理和最小数据结构。
先不要修改代码。
```
