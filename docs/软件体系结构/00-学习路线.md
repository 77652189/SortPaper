# 00-学习路线

你不需要按文件名从头读到尾。架构学习最怕“先补完所有理论再开始做项目”，因为真实项目会不断改变问题形状。更有效的方式是：先建立判断力，再用项目把知识钉牢。

## 路线 A：快速入门，先能和 Codex 说清楚

适合：你马上要让 Codex 帮你重构、拆模块、设计新功能。

阅读顺序：

1. `01-架构到底在解决什么问题.md`
2. `02-高内聚与低耦合.md`
3. `04-分层架构.md`
4. `11-AI辅助开发中的架构协作.md`
5. `CHECKLIST.md`

读完后你应该能做到：

- 判断一个功能是不是已经复杂到需要架构设计。
- 描述一个模块的职责、输入、输出和不应该做的事。
- 给 Codex 一个边界明确的重构任务。
- 看出“这个文件太大”背后真正的问题是什么。

## 路线 B：项目实战，围绕 RAG / Agent 设计

适合：你正在做论文解析、RAG 检索、Agent、长期记忆、风格学习。

阅读顺序：

1. `01-架构到底在解决什么问题.md`
2. `08-RAG系统架构.md`
3. `09-Agent系统架构.md`
4. `10-长期记忆与风格学习系统架构.md`
5. `06-六边形架构.md`
6. `12-架构设计实战模板.md`

读完后你应该能做到：

- 把 RAG 系统拆成解析、切分、清洗、索引、检索、生成等稳定模块。
- 判断 Agent 的工具调用、状态、记忆、计划循环是否混在一起。
- 为长期记忆写出“什么时候记、怎么查、怎么忘”的规则。
- 用模板写出一份可交给 Codex 实现的架构说明。

## 路线 C：系统补课，补齐常见架构模式

适合：你想系统理解 MVC、分层、事件驱动、六边形、插件化之间的关系。

阅读顺序：

1. `02-高内聚与低耦合.md`
2. `03-MVC与前后端职责边界.md`
3. `04-分层架构.md`
4. `05-事件驱动架构.md`
5. `06-六边形架构.md`
6. `07-插件化架构.md`

读完后你应该能做到：

- 知道每种架构模式在解决哪类痛点。
- 不会为了“显得专业”而过度设计。
- 能解释一个模块为什么放在这一层，而不是另一层。
- 能为解析器、检索器、Agent 工具、记忆策略设计扩展点。

## 必学与略读

必学：

- `01`：没有架构判断力，后面模式会变成术语。
- `02`：高内聚和低耦合是所有拆分的底层标准。
- `04`：分层架构是中大型项目最常用的稳定骨架。
- `06`：六边形架构能保护核心业务不被外部技术绑死。
- `08`、`09`、`10`：直接对应你的项目方向。
- `11`、`12`：把学习结果转成 Codex 可执行任务。

可以略读：

- `03`：如果项目没有明显 UI 或 API 展示层，可以先看模板部分。
- `05`：如果当前项目没有异步流程、任务队列、状态流转，可以先略读。
- `07`：如果系统短期内不需要多种解析器、检索器、工具或策略，可以先略读。

## 建议节奏

第一天：读 `01`、`02`、`04`，拿当前项目做一次边界检查。

第二天：读 `08`、`09`，把 RAG 和 Agent 的模块图画出来。

第三天：读 `10`、`11`、`12`，把一个真实改造任务写成 Codex 可执行说明。

之后每次新增功能前，先打开 `CHECKLIST.md`，用 10 分钟确认边界。这个小动作能省掉后面很多“越改越乱”的成本。
