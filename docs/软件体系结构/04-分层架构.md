# 04-分层架构

分层架构解决的是一个很朴素的问题：系统里有些代码靠近用户，有些代码靠近业务规则，有些代码靠近数据库和外部服务。它们如果没有方向感，就会互相调用，最后变成一团。

中大型 AI 项目尤其需要分层，因为它们通常同时有 UI、任务编排、领域规则、LLM、向量库、文件系统、缓存和监控。

## 为什么需要它

分层不是为了目录好看，而是为了控制依赖方向。

一个 RAG 系统里，`chunk 是否可入库` 是业务规则；Qdrant 只是存储实现。业务规则不应该散落在 Qdrant 调用代码里，否则以后换存储、换 payload、换入库策略都会互相影响。

一个 Agent 系统里，`工具执行失败后是否重试` 是应用流程规则；OpenAI SDK 或 LangGraph 只是实现方式。重试策略不应该被写死在某个 SDK 回调里。

## 常用四层

```text
入口层
  UI / API / CLI / batch command

应用层
  编排业务流程：解析、入库、检索、Agent run

领域层
  稳定业务概念和规则：Paper、Chunk、Memory、ToolResult

基础设施层
  Qdrant、LLM SDK、文件系统、PDF parser、缓存、外部 API
```

依赖方向应该大致是：

```text
入口层 -> 应用层 -> 领域层
应用层 -> 基础设施接口
基础设施层 -> 实现接口
```

领域层不要反过来 import UI、数据库或 SDK。

## 什么时候需要

需要：

- 系统已经有 UI、批处理、测试等多个入口。
- 同一个业务流程依赖多个外部技术。
- 想让 Codex 改某层时，不影响其他层。
- 想为核心逻辑写不依赖真实外部服务的测试。

暂时可以简单：

- 原型只有一个入口，一个外部服务。
- 领域规则还没稳定，过早分层会让迭代变慢。
- 项目很小，清晰函数比复杂目录更重要。

## 例子：论文处理 pipeline

坏设计：

```text
app.py
  Streamlit 控件
  PDF 解析
  chunk 合并
  LLM Judge prompt
  Qdrant 写入
  质量分析
  表格展示
```

这个结构里入口层成了系统中心。后续所有变化都往页面文件里堆。

更好的分层：

```text
入口层:
  Streamlit 页面、批量导入命令

应用层:
  PaperIngestionService
  PreviewPipeline
  QualityEnrichmentService

领域层:
  PaperId
  LayoutChunk
  ChunkVerdict
  StorePolicy

基础设施层:
  PdfParserAdapter
  LlmJudgeAdapter
  QdrantChunkRepository
```

入口层只问应用层：“请帮我预览这篇论文”或“请把解析结果入库”。它不需要知道 Qdrant point id 怎么生成。

## 例子：长期记忆系统

坏设计：

```text
chat_handler():
  判断是否要记忆
  生成记忆摘要
  写向量库
  查历史偏好
  改 prompt
```

更好的分层：

```text
入口层:
  Chat API / UI

应用层:
  ConversationService
  MemoryUpdateService

领域层:
  MemoryCandidate
  MemoryPolicy
  UserPreference
  StyleSample

基础设施层:
  VectorMemoryStore
  LlmSummarizer
```

这样你可以单独测试 `MemoryPolicy`：什么内容值得记，什么内容不值得记。

## 坏设计长什么样

- 领域对象里直接调用数据库。
- UI 文件里拼 LLM prompt。
- 基础设施层决定业务规则，比如“score 小于多少不入库”。
- 应用层返回 UI 组件，而不是业务结果。
- 多个入口复制同一套业务流程。

## 更好的拆法

- 入口层只处理输入输出，不承载核心规则。
- 应用层负责串流程，不写太多底层细节。
- 领域层保留稳定概念和判断规则。
- 基础设施层处理外部技术差异。
- 跨层通信用明确的数据结构，不靠散装字典。

## 可执行产物：分层依赖检查表

```markdown
## 分层检查

### 入口层
- 是否只负责输入、输出、用户动作？
- 是否直接访问数据库或 LLM SDK？

### 应用层
- 是否负责流程编排？
- 是否把领域判断委托给领域层？
- 是否通过接口调用外部服务？

### 领域层
- 是否不依赖 UI、数据库、SDK？
- 是否保存稳定业务概念和规则？

### 基础设施层
- 是否只实现外部技术细节？
- 是否没有偷偷决定核心业务规则？

### 反向依赖
- 领域层是否 import 了 Streamlit、Qdrant、OpenAI SDK？
- 基础设施层是否被 UI 直接调用？
```

## 给 Codex 的提示词

```text
请按入口层、应用层、领域层、基础设施层审查当前代码。
请指出每个主要函数现在属于哪一层，以及是否存在反向依赖。
要求输出最小重构顺序：先移动哪类逻辑，保持哪些公开入口不变。
先不要修改代码。
```
