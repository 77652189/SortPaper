# 软件体系结构检查清单

这个清单用于两件事：学习复盘，以及真实项目改造前的架构评审。不要把它当考试题。它的作用是帮你发现“哪里需要先想清楚，再让 Codex 动手”。

## 1. 架构必要性

- 当前功能是否超过 5 个处理步骤？
- 是否依赖 2 个以上外部技术，比如 LLM、向量库、PDF parser、数据库、UI？
- 是否有 preview、full pipeline、batch、agent 等多种模式？
- 是否存在失败重试、降级、跳过、恢复？
- 是否计划让 Codex 做跨文件改动？

判断：

- 0-2 项为“是”：保持简单，写清楚函数边界即可。
- 3-4 项为“是”：需要模块拆分和数据流说明。
- 5 项以上为“是”：需要接口契约、测试计划和分层边界。

## 2. 高内聚与低耦合

- 一个模块是否只有一个主要职责？
- 一个模块是否只有一类主要变化原因？
- 是否存在 `parse_and_store_and_render` 这类混合函数？
- 是否通过大字典传递隐式字段？
- 是否有多个模块重复同一业务规则？
- 测试某个模块时是否必须启动 UI 或真实外部服务？

通过标准：

- 能用一句话说清模块负责什么。
- 能说清模块不负责什么。
- 能单独测试核心规则。

## 3. MVC 与入口边界

- UI 是否直接访问数据库、向量库或 LLM SDK？
- 页面函数是否包含业务判断规则？
- 业务服务是否返回 UI 组件或 HTML？
- UI 状态和业务状态是否混在一起？
- 同一业务流程是否能被 UI、批处理、测试复用？

通过标准：

- View 只展示和收集输入。
- Controller 只处理用户动作和调用应用服务。
- 业务逻辑不依赖具体 UI 框架。

## 4. 分层架构

- 入口层是否只负责输入输出？
- 应用层是否负责编排流程，而不是写底层细节？
- 领域层是否不依赖 UI、数据库、SDK？
- 基础设施层是否没有偷偷决定核心业务规则？
- 是否存在反向依赖？

通过标准：

- 依赖方向清楚。
- 核心规则可以脱离外部服务测试。
- 外部技术变化不会扩散到全部业务代码。

## 5. 事件驱动

- 是否有一个步骤完成后触发多个后续动作？
- 后续动作是否可以异步执行？
- 是否需要记录状态变化？
- 事件名是否表示已经发生的事实？
- 事件 payload 是否小而稳定？
- 消费者失败是否有记录和重试策略？

通过标准：

- 事件用过去式命名，如 `PaperParsed`。
- 主流程不需要知道所有消费者。
- 事件失败不会无声丢失。

## 6. 六边形架构

- 核心业务是否直接 import Qdrant、OpenAI SDK、Streamlit？
- 是否为检索、存储、LLM、记忆定义了 Port？
- Adapter 是否负责外部格式和内部对象转换？
- 测试是否可以使用内存实现或 fake 实现？
- 替换外部技术是否只改 Adapter？

通过标准：

- 核心业务依赖能力接口，不依赖具体 SDK。
- 外部返回对象不会在系统内部到处流动。

## 7. 插件化

- 同一能力是否已经有 2 种以上实现？
- 主流程是否有大量类型分支？
- 新增解析器、检索器、工具或策略是否必须改核心流程？
- 插件是否有统一输入输出和错误格式？
- 配置是否集中管理？

通过标准：

- 新增插件主要是新增实现和注册。
- 主流程只依赖插件协议。

## 8. RAG 系统

- 是否保留 `raw_content` 和来源 metadata？
- chunk 是否有稳定 ID？
- 入库规则是否集中管理？
- 检索结果是否结构化返回 sources？
- rerank 和 context builder 是否独立？
- 回答是否记录使用了哪些 chunk？
- 是否能区分解析问题、检索问题和生成问题？

通过标准：

- RAG 每个阶段的输入输出清楚。
- 回答可以溯源。
- 失败能定位到具体阶段。

## 9. Agent 系统

- Agent 是否有结构化状态？
- 工具是否有统一协议和 schema？
- 是否有最大步数和停止条件？
- 工具失败是否标准化？
- 是否记录每一步 trace？
- 长期记忆是否经过过滤后再进入 Agent？

通过标准：

- 可以回放 Agent 的执行过程。
- 工具可以单独测试。
- Agent 不会无限循环或无边界调用工具。

## 10. 长期记忆与风格学习

- 是否区分事实、偏好、项目、交互、风格样本？
- 写入是否经过 MemoryPolicy？
- 检索结果是否带类型、来源、时间、置信度？
- 是否支持过期、删除、合并、冲突处理？
- 风格样本是否和事实记忆分开？
- Agent 是否记录本轮使用了哪些记忆？

通过标准：

- 不会无条件保存聊天记录。
- 不会把风格当事实。
- 可以解释某条记忆为什么被使用。

## 11. Codex 协作准备

在让 Codex 实现前，确认任务说明包含：

- 目标。
- 为什么需要。
- 保持不变的公开入口。
- 允许改变的内部边界。
- 不允许改变的数据字段语义。
- 测试计划。
- 手动验证方式。
- 完成后的总结要求。

推荐提示词：

```text
请先按架构检查清单审查当前实现。
只输出问题和最小改造方案，不要修改代码。
重点关注职责边界、依赖方向、接口契约和测试风险。
```

## 12. 最后判断

如果你不能回答下面 5 个问题，就先不要让 Codex 大改：

1. 这个功能的核心输入和输出是什么？
2. 哪些接口必须保持兼容？
3. 哪些规则属于业务，哪些只是外部技术细节？
4. 失败时应该怎么表现？
5. 做完后用什么测试证明没坏？

能回答这 5 个问题，AI 辅助开发就会稳很多。
