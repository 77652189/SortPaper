# 05-事件驱动架构

事件驱动不是为了把系统变酷。它解决的是长流程里的一个痛点：一件事发生后，可能有多个后续动作，但这些动作不应该全部写死在主流程里。

例如论文解析完成后，你可能要入库、生成摘要、更新质量报告、刷新 UI、通知 Agent 索引可用。把这些都写在 `parse_pdf()` 后面，主流程会越来越臃肿。

## 为什么需要它

事件的意义是把“已经发生的事实”和“谁要对此做反应”分开。

```text
事实：PaperParsed
反应：StoreChunks、EvaluateQuality、BuildSearchIndex、UpdateUI
```

主流程只发布事实，不需要知道所有后续消费者。

在 AI 系统里，这对异步任务尤其有用。LLM 调用慢、向量库写入可能失败、图片解析可能排队。如果所有事情同步串在一起，用户会等很久，失败恢复也困难。

## 什么时候需要

需要：

- 一个步骤完成后，会触发多个后续动作。
- 后续动作可以异步执行，不必阻塞主流程。
- 你需要记录状态变化，比如 parsed、stored、enriched、indexed。
- 不同模块需要订阅同一种业务事实。

暂时不要用：

- 流程很短，直接调用更清楚。
- 事件没有可靠日志，失败后无法追踪。
- 团队还没定义事件命名和 payload，随手发事件会更乱。
- 只是为了避免函数调用而上事件系统。

## 例子：论文解析状态流转

坏设计：

```text
run_pipeline():
  parse_text()
  parse_table()
  parse_image()
  store_chunks()
  enrich_quality()
  refresh_ui()
  write_report()
```

这里每新增一个后续动作，都要改主流程。

更好的事件设计：

```text
PaperParsed
ChunksJudged
ChunksStored
PaperEnriched
```

消费者：

```text
on ChunksStored:
  update_search_index
  schedule_quality_enrichment
  record_ingestion_metrics
```

这让主流程关注核心阶段，扩展动作通过订阅加入。

## 例子：长期记忆与风格学习

一次对话结束后，可能发生这些事：

- 从对话中提取记忆候选。
- 判断是否写入长期记忆。
- 提取用户偏好的表达方式。
- 更新风格样本库。
- 记录哪些记忆被回答使用过。

如果这些都写在 `answer_user()` 里，问答逻辑会变重。

事件化后可以是：

```text
AnswerGenerated
MemoryCandidateFound
MemoryUpdated
StyleLearned
```

问答服务只负责生成回答并发布 `AnswerGenerated`。记忆模块和风格学习模块各自订阅。

## 坏设计长什么样

- 事件名很像命令，比如 `DoStoreChunks`，让人分不清事实和动作。
- 事件 payload 是一个巨大字典，没人知道字段契约。
- 事件消费者偷偷修改主流程依赖的数据。
- 事件失败没有重试、没有日志、没有幂等。
- 同一个事件在不同地方含义不同。

## 更好的拆法

- 事件命名使用过去式：`PaperParsed`、`ChunksStored`、`MemoryUpdated`。
- 事件只描述事实，不命令别人做什么。
- payload 保持小而稳定，只放消费者需要的核心标识和摘要。
- 消费者失败要可记录、可重试、可跳过。
- 对会重复收到的事件，消费者要尽量幂等。
